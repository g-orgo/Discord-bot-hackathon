## Issues Found

### CRITICAL

#### C1 — `commands.js`: `TEST_COMMAND` and `CHALLENGE_COMMAND` undefined at runtime
**File:** `raptor-chatbot/commands.js`
**Status:** ✅ Fixed

A code formatter removed the `TEST_COMMAND` and `CHALLENGE_COMMAND` constant definitions (and their helper imports) while preserving their references in `ALL_COMMANDS`. This caused a `ReferenceError` crash on `yarn register`.

**Fix applied:**
- Restored `TEST_COMMAND` definition
- Removed `CHALLENGE_COMMAND` from `ALL_COMMANDS` — the slash command handler is commented out in `app.js`, so registering it would have caused Discord to invoke a command the bot doesn't handle
- `createCommandChoices()` and `getRPSChoices` import were not restored — not needed without `CHALLENGE_COMMAND`

---

### MAJOR

#### M1 — `app.md` context file is stale — missing `/ask` and `/logchannel` commands
**File:** `raptor-chatbot/.claude/context/app.md`

The APPLICATION_COMMAND dispatch table only documents `test` and `challenge`. `/logchannel` and `/ask` are completely absent. This causes incorrect context to be loaded by the dev skill.

**Recommendation:** Update `app.md` to reflect the current command set.

#### M2 — `commands.md` context file is stale — outdated command list
**File:** `raptor-chatbot/.claude/context/commands.md`

Only documents `/test` and `/challenge`. Neither `/logchannel`, `/ask`, nor the removal of `CHALLENGE_COMMAND` are reflected.

**Recommendation:** Rewrite `commands.md` to match current state of `commands.js`.

#### M3 — `GET /system-prompt` is publicly accessible
**File:** `raptor-llm/main.py`

The system prompt is exposed via `GET /system-prompt` with no authentication. If the server is publicly accessible, this leaks internal prompt engineering.

**Recommendation:** Add API key or restrict to localhost unless public access is intentional. Already documented as known issue in context.

### MINOR

#### m1 — FastAPI `description` was stale ("Hello World LLM API using Ollama")
**File:** `raptor-llm/main.py`
**Status:** ✅ Fixed
Updated to `"LLM API server for raptor-chatbot — powered by Ollama"`.

#### m2 — `update_system_prompt` uses `global` — not thread-safe in multi-worker deployments
**File:** `raptor-llm/main.py`
For single-worker dev use this is fine, but a multi-worker production deployment (e.g. `--workers 4`) would have divergent SYSTEM_PROMPT state per worker.
**Recommendation:** Acceptable for current use. Note if deploying with multiple workers.

#### m3 — PATCH in `select_choice` handler doesn't include `IS_COMPONENTS_V2` flag
**File:** `raptor-chatbot/app.js` (line ~210)
The ephemeral message PATCH body omits `flags: IS_COMPONENTS_V2`. The original message already has this flag set, so it likely works, but is inconsistent with convention.
**Recommendation:** Add `flags: InteractionResponseFlags.IS_COMPONENTS_V2` to the PATCH body for consistency.

#### m4 — User-supplied message echoed in public Discord response without sanitization
**File:** `raptor-chatbot/app.js` — `/ask` handler
`**Você:** ${message}` renders user input as Discord markdown. A malicious user could craft a message with markdown that renders unexpectedly.
**Recommendation:** Low risk in Discord (no script injection), but worth noting if this is used in wider contexts.
