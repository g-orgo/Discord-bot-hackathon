# Audit Report — 2026-04-06 (run 2)

Second cross-stack audit. One critical defect found and fixed in `raptor-chatbot`: `game.js` is missing from the filesystem while `app.js` still called its functions, guaranteeing a `ReferenceError` crash on any component interaction. Three minor issues documented. Both stacks are otherwise clean.

## Issues Found

### CRITICAL

#### C1 — `game.js` missing; component handlers crash on call
**File:** `raptor-chatbot/app.js`
**Status:** ✅ Fixed

`game.js` no longer exists on disk. `app.js` was calling `getShuffledOptions()` (in `accept_button_*` handler) and `getResult()` (in `select_choice_*` handler) without importing them. Any click on a stale challenge button would throw `ReferenceError: getShuffledOptions is not defined` and crash the handler.

Additionally, `activeGames` was declared but could never be populated (challenge command is not registered), making it dead state.

**Fix applied:**
- Replaced both component handlers with a single graceful "Este desafio já não está disponível." ephemeral response
- Removed the unused `activeGames` object
- Added a comment in the import section documenting that `game.js` was removed

---

### MINOR

#### m1 — `utils.js` exports unused `capitalize` function
**File:** `raptor-chatbot/utils.js`

`capitalize` was used by `commands.js` to format RPS choice labels. Now that the challenge command and `game.js` are removed, `capitalize` has no consumers in the codebase. It remains exported but is never called.

**Recommendation:** Remove `capitalize` export from `utils.js` when the challenge feature is permanently dropped. Left in place this audit as it is harmless and could be reused if the feature returns.

#### m2 — `/logchannel` returns success message even when Discord API fetch fails
**File:** `raptor-chatbot/app.js`

The `try/catch` in the `/logchannel` handler wraps only the fetch — the `return res.send(...)` with "As mensagens recentes do canal foram logadas no console" always executes, even if the fetch threw. The user sees a success message while the actual log silently failed.

**Recommendation:** Move the `return res.send(...)` inside the try block and send an error message in the catch, or at minimum change the success text to "Tentativa de log realizada — verifique o console."

#### m3 — `OLLAMA_URL` is hardcoded with no env var override
**File:** `raptor-llm/main.py`

`OLLAMA_URL = "http://localhost:11434/api/generate"` is a hardcoded constant. If Ollama runs on a different host (e.g., Docker, remote server), there is no way to override it without editing the source.

**Recommendation:** `OLLAMA_URL = os.getenv("OLLAMA_URL", "http://localhost:11434/api/generate")`
